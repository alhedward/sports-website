LROC Handover — v3.4.7 Article Renderer Whitespace Reduction
Generated: 2026-06-14 AEST

Baseline
- v3.4.6 fixed the major issue where TinyMCE article pages were placed across the correct Issue Builder pages and the preview no longer upscaled above the page box.
- User confirmed all content now flows across pages, but the next blocker is large blank white space on non-final pages.
- The working assumption is that many gaps are caused by figure/image blocks being treated as indivisible. When a figure does not fit at the end of a page/column, the previous renderer moved the whole block to the next page and left the bottom of the current page empty.

What changed
- mag340McePaginateHtmlByBlocks now has v3.4.7 whitespace-reduction behaviour.
- Media-like blocks can be marked shrink-to-fill with class mag347-shrunk-media / mag347-shrunk-image when shrinking lets the block use remaining column/page space.
- If a media block still cannot sensibly fit, the paginator looks ahead for a few small text blocks and pulls them forward, then places the media on the next page. This deliberately moves image position slightly in the final render to reduce avoidable white space.
- Renderer diagnostics now report:
  - rendererVersion: 3.4.7
  - whiteSpaceAdjustments
  - pulledForwardTextBlocks
  - shrunkMediaBlocks
- CSS support was added for mag347 shrink-to-fill media in both measurement and rendered proof views.
- Pagination mode is now column_width_block_aware_whitespace_reduce_v347.

Important behaviour notes
- This is not yet a full constraint solver. It is a practical magazine paginator pass focused on the common problem: images/figures creating bottom-of-page holes.
- Image blocks are still preserved as visual units where possible; they are shrunk before being split/cropped.
- The renderer may move following text ahead of an image to fill a page. That is intentional and matches the user's direction that image location can move in the final render/preview to reduce blank areas.
- The final article page may still have white space because there may simply be no more content.

Continue from here
- If non-final pages still have large white areas, inspect layoutSlots[0].mcePaginationDiagnostics in the browser console.
- If whiteSpaceAdjustments remains 0 on pages with obvious image-caused gaps, expand the lookahead window or lower the remaining-space threshold in mag340McePaginateHtmlByBlocks.
- If images become too small, raise the minimum image cap inside mag347CompactMediaBlock.
- A future stronger version should expose image priority/placement controls in Article Library: inline, top of next column, half-column, full-width, or optional shrink.
